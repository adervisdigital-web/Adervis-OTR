# TG-сайдбар: виртуализация списка лидов

**Дата:** 2026-07-01
**Статус:** Одобрено, готово к написанию плана реализации

## Контекст

Roadmap v4 (Пилар 1, пп.1-2) отметил `renderTgSidebar()` как узкое место: функция перестраивает весь DOM сайдбара диалогов при каждом вызове (смена данных, фильтр поиска, входящее сообщение), и при росте базы лидов до сотен записей это станет заметно медленнее.

Эта сессия закрывает только сайдбар (`#tgLeadList`), не таблицу лидов (`renderTable()`) — они разбиты на два отдельных подпроекта, потому что архитектурно различны:
- Сайдбар уже находится в собственном `overflow-y:auto` контейнере фиксированной высоты (`#tgLeadList { flex: 1; overflow-y: auto; }`) — виртуализация ложится без структурных изменений разметки.
- Таблица скроллится вместе со всей страницей (window scroll) — виртуализация потребует обернуть её в отдельный скролл-контейнер, это отдельная, более рискованная задача.

Модуляризация `index.html` на ES-модули (тоже из Roadmap Пилара 1) намеренно исключена из этой работы — она отдельно рискованна из-за 157 инлайн `onclick="..."` атрибутов по всему файлу, которые полагаются на глобальную область видимости функций и сломаются при переходе на `type="module"` без отдельного продуманного решения.

## Что усложняет задачу: сайдбар не плоский список

`renderTgSidebar()` сейчас строит не единый список, а несколько частей подряд:
1. Баннер «Лучшее время для касания» (опционально, 0 или 1 штука)
2. Заголовок секции «🔥 Требуют действия» (всегда, даже если 0 срочных) + сами срочные лиды
3. Заголовок «💬 В диалоге» (если есть) + лиды
4. Заголовок «📋 Все лиды» (если есть) + лиды
5. Свёрнутая по умолчанию секция «Закрытые» (раскрывается кнопкой, `display:none` пока не открыта)

## Архитектура

**Секция «Закрытые» не виртуализуется.** Она свёрнута по умолчанию и раскрывается вручную кнопкой — маловероятно, что менеджер держит открытыми одновременно сотни закрытых/отказных лидов на экране. Рендерится как сейчас, без изменений.

**Активные секции (баннер + 🔥/💬/📋) схлопываются в один плоский массив «строк».** Каждая строка — дескриптор одного из двух типов:
- `{ type: 'header', html, height }` — баннер или заголовок секции
- `{ type: 'lead', lead, group, height }` — один лид (group нужен `renderTgLeadItem` как сейчас для стилизации по типу секции)

Массив строится в том же порядке, что и сейчас (urgent → dialog → new), с накопленными смещениями (`offsetTop` каждой строки = сумма высот всех предыдущих).

**Измерение высоты строки — из реального первого рендера, не хардкод.** При самом первом вызове `renderTgSidebar()` после загрузки страницы сайдбар рендерится как обычно (без окна, весь список), после чего через `getBoundingClientRect()` снимается реальная высота одного `.sb-section-lbl` (заголовок) и одного `.tg-lead-item` (лид) — эти два числа кешируются в модульных переменных `_tgHeaderH`/`_tgLeadH` и используются для всех последующих (уже оконных) рендеров. Если на странице пока нет ни одного заголовка/лида для измерения (пустой список) — используются консервативные константы-заглушки (32px/68px, подобранные по текущим CSS-паддингам), которые переизмеряются при следующей возможности.

**Два уровня рендеринга:**
1. `renderTgSidebar(selectId)` — существующая точка входа, вызывается при смене данных/фильтра/поиска (как сейчас). Пересобирает плоский массив строк и их смещения (дорогая часть — фильтрация/сортировка/группировка `leads`), сохраняет их в модульных переменных, затем вызывает `renderTgSidebarVisibleWindow()`.
2. `renderTgSidebarVisibleWindow()` — новая, дешёвая функция. Читает `list.scrollTop` и высоту вьюпорта, по закешированным смещениям строк находит видимый диапазон ± overscan (8 строк сверху и снизу), строит HTML только для этого диапазона, оборачивает в два spacer-div (высота = сумма пропущенных строк сверху / снизу), записывает в `list.innerHTML`.

**Обработчик скролла** вешается на `#tgLeadList` один раз (не при каждом рендере) — при каждом событии `scroll` планирует вызов `renderTgSidebarVisibleWindow()` через `requestAnimationFrame` (не чаще одного раза за кадр), не трогая дорогую пересборку данных из пункта 1.

## Совместимость с существующим кодом

`updateTgSidebarItem(leadId)` и `filterTgSidebar(query)` продолжают вызывать `renderTgSidebar(currentChatLeadId)` как сейчас — снаружи поведение точки входа не меняется, остальной код приложения не трогаем.

## Граничные случаи

- **Пульс-анимация на входящее сообщение** (`document.querySelector('[data-lead-id="..."]')` + добавление CSS-класса) сейчас ищет элемент напрямую в DOM. Если лид вне текущего окна рендера, элемента в DOM не будет. Решение: перед пульсом проверяем, есть ли лид в видимом диапазоне; если нет — сначала скроллим `#tgLeadList` к вычисленной позиции этого лида (`offsetTop` из закешированного массива строк), затем даём кадру отрендериться, затем применяем пульс.
- **Выбор лида через поиск/палитру команд**, когда он вне текущего окна — аналогично, скроллим к вычисленной позиции перед тем как подсвечивать `.active`.
- **Пустой список / список из 1-2 строк** — короче вьюпорта, окно = весь список, spacer-высоты = 0, разницы с обычным рендером нет.
- **Быстрый scroll/momentum на мобильных** — overscan-буфер 8 строк снижает риск пустых мельканий между кадрами.
- **Смена темы/масштаба страницы после первого измерения** — высоты строк не переизмеряются автоматически. Если это станет проблемой на практике, тривиально добавить переизмерение при смене `data-theme`, но заранее не проектируем — не наблюдаемая сейчас проблема.

## Тестирование

В проекте нет автотестов ни для одного JS-файла — проверка вручную:
1. Временно засеять локально ~300 синтетических лидов (или продублировать существующие через консоль браузера) — скролл сайдбара должен оставаться плавным, DOM не должен содержать сотни `.tg-lead-item` одновременно (проверить через DevTools)
2. Входящее сообщение от VK/TG для лида, находящегося вне текущего окна прокрутки — сайдбар должен проскроллиться к нему и показать пульс
3. Выбор лида через `Ctrl+K` палитру или поиск, когда он не в видимой области — сайдбар должен проскроллиться и подсветить его
4. Раскрытие секции «Закрытые» — работает как раньше, без виртуализации
5. Смена фильтра поиска — список корректно пересобирается, скролл сбрасывается к началу
6. Обычная работа со списком из нескольких лидов (текущий типичный объём) — визуально неотличимо от поведения до изменений

## Вне рамок (Out of Scope)

- Виртуализация таблицы лидов (`renderTable()`) — отдельный подпроект, требует обёртывания в собственный скролл-контейнер
- Модуляризация `index.html` на ES-модули — отдельный подпроект, блокируется 157 инлайн `onclick` атрибутами
- Diff-рендер вместо full re-render (Roadmap Пилар 1 п.3) — виртуализация решает проблему количества DOM-узлов, но `renderTgSidebar()` при смене данных всё ещё пересобирает HTML видимого окна целиком, а не точечно
- Виртуализация свёрнутой секции «Закрытые»
