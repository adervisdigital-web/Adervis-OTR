# ADERVIS OTR — Design Spec: Product Improvements
**Дата:** 2026-06-30  
**Статус:** Approved by user  

---

## Контекст

Приложение OTR — это CRM для холодных продаж ADERVIS (видеопродакшн). Стек: single-file HTML (8681 строк, Vanilla JS), Supabase backend, Edge Functions на Deno/TypeScript. Активные каналы: TG Bot, VK Bot, Instagram (pending credentials).

Этот спек покрывает 10 улучшений в порядке реализации: от высокоприоритетных к долгосрочным.

---

## Улучшение 1 — VK AI-ответ на свободный текст

**Проблема:** VK-бот отвечает только на первое сообщение (авто-ответ + кнопки) и на нажатия кнопок клавиатуры. Любой свободный текст от клиента игнорируется.

**Решение:** В `supabase/functions/vk-webhook/index.ts` после сохранения входящего сообщения добавить вызов Gemini API (аналогично tg-webhook). Если сообщение НЕ является ответом бота (fromClient=true), генерировать AI-ответ на основе:
- Контекста бизнеса (`bizType` лида)
- Истории последних 6 сообщений (messages[])
- Промпта из `workspace_settings.tg_ai_prompt` (переиспользовать)

**Архитектура:**
```
VK message_new
  → if isFirstMessage → авто-ответ + кнопки (уже есть)
  → if кнопка (payload) → обработка кнопки (уже есть)
  → if свободный текст + НЕ первое → Gemini → vkApiSend()
```

**Изменения:**
- `vk-webhook/index.ts`: добавить блок `else if (isTextMessage && !isFirstMessage)` → вызов Gemini
- Дублировать функцию построения промпта из tg-webhook (shared utils между Edge Functions в Deno/Supabase нетривиальны — дублирование проще)
- Gemini ключ уже установлен в secrets

**Не делать:** бриф через VK (это п.2, отдельно). AI отвечает только на свободный текст.

## Улучшение 2 — VK мини-бриф (3 вопроса)

**Проблема:** В TG-боте — полный бриф из 6 вопросов. В VK после нажатия "📋 Оставить заявку" — только запрос контакта. Лиды из VK приходят без информации о бизнесе/бюджете.

**Решение:** Реализовать 3-шаговый бриф в VK-боте:
1. "Расскажите про ваш бизнес — чем занимаетесь?"
2. "Как вас зовут и как удобно связаться?"
3. По завершении → push-уведомление менеджеру + status=2

**Хранение состояния:** поле `vk_brief_step INT` в таблице `leads` (NULL = не в брифе, 1/2/3 = текущий шаг). Либо через `tg_state JSONB` добавить `vk_brief_step`.

**Изменения:**
- Миграция: добавить `vk_brief_step INT` в `leads` (или расширить tg_state)
- `vk-webhook/index.ts`: добавить FSM для брифа (аналогично TG, но только 3 вопроса)
- Push-уведомление при завершении брифа (переиспользовать существующую логику)

## Улучшение 3 — Потенциальная выручка в воронке

**Проблема:** Воронка показывает только количество лидов. Менеджер не видит денежный потенциал pipeline.

**Решение:** Парсить ответ на вопрос брифа о бюджете и показывать суммарный потенциал под воронкой.

**Парсинг бюджета:**
- В TG брифе есть кнопки: "до 30к", "30-80к", "80-200к", "200к+"
- Маппинг: до30к=15000, 30-80к=55000, 80-200к=140000, 200к+=250000
- Из messages[] найти сообщение с `type='brief_answer'` и ключевым словом бюджета
- Сохранять `deal_budget INT` в leads при заполнении брифа

**Отображение:** В `renderDashFunnel()` и `renderFunnelHero()` добавить строку:
```
💰 Потенциал pipeline: 3 лида × ~55к = 165 000 ₽
```
Показывать только для лидов в статусах 1, 2, 3 (не архив, не отказ).

**Изменения:**
- Миграция: `deal_budget INT` в leads (если нет — добавить)
- tg-webhook: при `brief_complete` сохранять deal_budget
- vk-webhook: то же при завершении мини-брифа (п.2)
- index.html: `renderDashFunnel()` / `renderFunnelHero()` — добавить расчёт и вывод

## Улучшение 4 — Дедупликация VK ↔ TG

**Проблема:** Один клиент пишет и в VK, и в TG — создаются 2 разных лида, менеджер видит дублей.

**Решение:** При завершении брифа (когда известен телефон/username) — поиск существующего лида в workspace по совпадению контакта.

**Логика:**
1. Завершён бриф (получен контакт) → нормализовать контакт (убрать +7/8, пробелы)
2. Поиск в `leads` по `contact` (ILIKE + нормализация)
3. Если найден → merge: добавить новые сообщения к существующему лиду, обновить `vk_peer_id` или `tg_chat_id`
4. Если не найден → создать новый лид как обычно

**Где реализовать:** В Edge Function при `brief_complete` (и tg-webhook, и vk-webhook).

**Изменения:**
- Добавить функцию `findDuplicateLead(supabase, workspaceId, contact)` в оба webhook
- При merge: `UPDATE leads SET messages = merged_messages, vk_peer_id = ..., contact = ...`
- Не трогать UI OTR — изменения только на бэкенде

## Улучшение 5 — Метрика "время ответа" в карточке лида

**Проблема:** Менеджер не знает насколько "горячий" лид — нет данных о скорости реакции клиента.

**Решение:** Вычислять среднее время ответа клиента из messages[] и показывать в Lead Drawer.

**Вычисление (фронтенд, без бэкенда):**
```js
// Пары: [msg от менеджера → следующий msg от клиента]
// avg_response = mean(client_msg.date - prev_manager_msg.date)
```

**Отображение в Lead Drawer:** строка под контактом:
```
⏱ Отвечает в среднем за 2ч 15м
```
Если меньше 2 сообщений — не показывать. Если > 48ч — подсветить как "холодный".

**Изменения:**
- index.html: добавить функцию `calcAvgResponseTime(messages)` 
- Вставить в `renderLeadDrawer()` (или аналогичную функцию отображения карточки)

## Улучшение 6 — Автоматический VK follow-up

**Проблема:** Edge Function `tg-reminder` отправляет follow-up TG-лидам через 24ч без ответа. VK-лиды без ответа остаются без напоминания.

**Решение:** Расширить `tg-reminder` (или создать `vk-reminder`) для VK-лидов.

**Логика:**
- Cron каждые 6ч (уже настроен для TG)
- Найти VK-лиды: `vk_peer_id IS NOT NULL`, `status IN (1,2)`, последнее сообщение от клиента > 24ч назад
- Отправить через `vkApiSend()`: текст follow-up из `workspace_settings.vk_welcome_text` или отдельное поле `vk_followup_text`
- Отметить в leads: `vk_followup_sent_at = NOW()` (чтобы не слать повторно)

**Изменения:**
- Миграция: `vk_followup_text TEXT` в `workspace_settings`, `vk_followup_sent_at TIMESTAMPTZ` в `leads`
- `tg-reminder/index.ts`: добавить блок для VK (или создать `vk-reminder/index.ts`)
- Настройки OTR: текстовое поле "Текст follow-up для VK" в секции VK

## Улучшение 7 — Быстрые ответы для VK-чата в OTR

**Проблема:** В TG-чате OTR под полем ввода есть кнопки быстрых фраз. В VK-режиме их нет — менеджер каждый раз печатает с нуля.

**Решение:** Добавить те же кнопки быстрых фраз под `#chatInputMain` когда активен VK-чат лида (`lead.vk_peer_id != null`).

**Быстрые фразы:** переиспользовать существующие из TG-режима (они уже в workspace_settings как `quick_replies`). Если поле не заполнено — показывать дефолтные:
- "Отлично! 🎬"
- "Уточните, пожалуйста"
- "Свяжусь сегодня 📞"

**Изменения:**
- index.html: в `renderChatHeader()` или `renderChatView()` — при `lead.vk_peer_id` показывать кнопки быстрых фраз
- Переиспользовать `insertTemplateToChat(text)` для вставки фразы в textarea

## Улучшение 8 — Ежедневная продуктивность менеджера

**Проблема:** Daily goal показывает только "добавлено X лидов". Менеджер не видит сколько сообщений написал сегодня.

**Решение:** Добавить в "Focus Today" панель счётчик отправленных сообщений за сегодня.

**Вычисление (фронтенд):**
```js
// Из всех leads.messages:
// fromClient=false, date >= startOfToday → count
```

**Отображение:** в блоке Focus Today рядом с daily goal:
```
✍️ Написано сегодня: 12 сообщений
```

**Изменения:**
- index.html: добавить функцию `countTodayMessages()` 
- Вставить в `updateDashboard()` → рендер Focus Today секции

## Улучшение 9 — Экспорт аудитории для таргета

**Проблема:** Лиды со статусом "Отказ" — готовая аудитория для ретаргета в VK/Instagram. Нет способа быстро её выгрузить.

**Решение:** Кнопка "Экспорт для таргета" в разделе фильтров (или в массовых действиях).

**Формат CSV:**
```
name;link;contact;bizType;platform
ООО Кафе Уют;https://vk.com/cafe_uyut;;кафе;vk
```

**Критерии:** `status=4` (Отказ) + `!archived_at`. Опционально — все статусы кроме Успех.

**Изменения:**
- index.html: добавить функцию `exportTargetAudience()` (аналогично `exportCsv()`)
- Кнопка в секции фильтров или в header таблицы рядом с "Экспорт CSV"

## Улучшение 10 — Рефакторинг: разбивка index.html на модули

**Проблема:** index.html = 8681 строк. При правке одной функции риск задеть другую. Браузер парсит весь файл. Сложно дебажить.

**Решение:** Минимальный рефакторинг без изменения поведения — вынести JavaScript в отдельные `<script src>` файлы через ES Modules или сборку.

**Подход (без build-step, максимально консервативно):**
- Оставить HTML-шаблоны и CSS в index.html
- Вынести JS в несколько файлов: `app-core.js`, `app-ui.js`, `app-db.js`, `app-bot.js`
- Подключить через `<script type="module" src="...">` 
- Локальная разработка: `python -m http.server` или `npx serve` (уже используется)
- PWA service worker — обновить список кешируемых файлов

**Решение по PWA/кешу:** Новые JS-файлы добавить в `sw.js` precache list. При деплое на GitHub Pages (текущий хостинг) модули работают без проблем — браузер поддерживает ES Modules нативно. Если что-то сломается — откатить и оставить inline.

**Изменения:**
- Создать `js/` директорию с 4 файлами
- index.html: заменить inline `<script>` на `<script type="module" src="js/app-core.js">`
- `sw.js`: добавить новые файлы в precache list
