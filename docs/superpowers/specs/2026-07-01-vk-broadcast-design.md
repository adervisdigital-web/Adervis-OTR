# VK Массовая рассылка — дизайн

**Дата:** 2026-07-01
**Статус:** Одобрено, готово к написанию плана реализации

## Контекст

Менеджер работает с личным VK-аккаунтом (не community-ботом) для части лидов. Периодически нужно разослать один и тот же текст (ледокол или follow-up) сразу нескольким лидам вместо ручного копирования в каждый диалог. Ограничение платформы: личный аккаунт, VK ограничивает ~30-50 новых диалогов в день — превышение рискует блокировкой аккаунта.

Брейнсторм проходил в две сессии (30 июня — 1 июля). Архитектура транспорта была одобрена ещё 1 июля: резолвить `screen_name → vk_peer_id`, затем последовательно слать с задержкой 12 сек между сообщениями. Эта сессия закрывает оставшуюся часть — UI и оставшиеся технические развилки (обработка ошибок, дневной лимит, история сообщений).

## Точка входа

Расширяем существующую панель массовых действий (Bulk Actions), а не тулбар и не отдельную страницу.

- Менеджер отмечает лидов чекбоксами в таблице, как сейчас для удаления
- В bulk-баре появляется кнопка **«📤 Разослать выбранным (N VK)»**, где N — сколько из отмеченных имеют VK-ссылку (`lead.link` похож на vk.com/... или уже есть `vkPeerId`)
- Если ни у одного из выбранных нет VK-ссылки — кнопка не показывается вовсе

**Почему не тулбар/не отдельная страница:** при дневном лимите 30-50 сообщений точный контроль получателей важнее скорости доступа, а чекбоксы — уже привычный паттерн в этом UI (используется для массового удаления). Отдельная страница добавила бы код ради редко используемой функции.

## Экран подтверждения

Открывается модалкой сразу по клику «Разослать выбранным» — без промежуточного шага выбора лидов (они уже выбраны чекбоксами, отдельный «шаг 1: лиды» из более раннего варианта дизайна избыточен).

Содержимое:
1. **Получатели** — список чипов, каждый лид помечен ✅ (peer_id есть или резолвится) либо ⚠️ с причиной («нет VK» / «не удалось резолвить»). Пропущенные лиды не участвуют в рассылке, но видны для прозрачности.
2. **Предупреждение о дневном лимите** (если применимо) — баннер вида «Сегодня уже отправлено N через VK, эта рассылка доведёт до M» при приближении к 30-50. Не блокирует запуск, только информирует — решение остаётся за менеджером.
3. **Шаблон** — выпадающий список из уже существующих шаблонов этапа «Ледокол»/follow-up (`scripts` таблица, stage 1) — отдельного набора «шаблонов для рассылки» не заводим, это тот же пул, что и в обычном диалоге.
4. **Предпросмотр** — текст выбранного шаблона после `substituteCta()` (те же плейсхолдеры `{call_link}`/`{brief_link}`, что и в ручной отправке)
5. **Оценка времени** — `~N × 12 сек`
6. Кнопка **«Начать рассылку →»**

### Резолв screen_name → peer_id

Часть лидов имеют VK-ссылку вида `vk.com/cafe_uyut` (screen_name), но не числовой `vkPeerId` (сейчас `extractVkPeerId()` вытаскивает id только если ссылка уже числовая, `vk.com/id12345`). При открытии экрана подтверждения для лидов без `vkPeerId`, но с похожей на VK ссылкой, вызывается новый edge function (аналог `vk-conversations`, но в обратную сторону) `vk-resolve-peer`, который через VK API `utils.resolveScreenName` личного аккаунта резолвит screen_name → numeric id. Успешно резолвленный `vkPeerId` сразу сохраняется на лида (`upsertLead`), чтобы не резолвить повторно при следующей рассылке или обычной отправке. Лиды, которые не удалось резолвить (приватный профиль, страница не существует), помечаются ⚠️ и исключаются из отправки.

## Экран отправки

Прогресс-бар сверху (проценты + таймер «осталось ~N сек») + список получателей под ним, каждый со статусом:
- ⬜ в очереди
- ⏳ ждёт/отправляется (текущий)
- ✅ отправлено
- ❌ ошибка (с кратким текстом причины при наведении)

Кнопка **«⏹ Остановить»** — прекращает отправку оставшимся в очереди; уже отправленные сообщения остаются отправленными, откат невозможен и не нужен.

**Почему бар+список, а не лог-поток:** целевой пользователь — менеджер продаж, не разработчик; иконки статуса считываются быстрее монотипного лога.

## Экран завершения

Сводка: «✅ N отправлено · ❌ K ошибок · ⚠️ M пропущено», список по каждому лиду с итоговым статусом, кнопка «Закрыть».

## Архитектура и компоненты

Всё в `index.html` (клиентская логика), плюс один новый edge function. Существующий `vk-send` не меняется — он уже принимает один `lead_id` за вызов, это то, что нужно для последовательной отправки с задержкой.

- `vk-resolve-peer` (новый edge function) — принимает список `{lead_id, screen_name}`, резолвит через VK API личного аккаунта (используется тот же токен/паттерн доступа, что в `vk-conversations`), возвращает `{lead_id, peer_id | error}` по каждому
- `openVkBroadcastModal(selectedLeadIds)` — рендерит экран подтверждения, вызывает резолв для лидов без `vkPeerId`
- `startVkBroadcast(recipients, templateText)` — асинхронный цикл: для каждого получателя — вызов `vk-send`, на успехе — `addMsg`-эквивалент (обычное исходящее сообщение в `lead.messages[]`, без спец-тега, как договорились), на ошибке — пометка ❌ и переход к следующему без остановки цикла; ожидание 12 сек перед каждым следующим вызовом (не перед первым); проверка флага остановки перед каждой итерацией
- `stopVkBroadcast()` — выставляет флаг, проверяемый в цикле
- Подсчёт дневного лимита — расширение существующей `countTodayMessages()` (добавлена в sprint 30 июня) фильтром по `lead.vkPeerId != null`, чтобы считать только VK-сообщения, а не все каналы

**Data flow:** Bulk selection (existing) → confirm modal (resolve + preview) → send loop (client-side setTimeout/await, browser вкладка должна оставаться открытой) → per-recipient `vk-send` call → `upsertLead` (message history + persisted peer_id) → live UI update по каждому получателю → summary screen.

## Обработка ошибок

- **Ошибка отправки одному лиду** (VK API error, rate limit) — помечается ❌ с причиной, цикл продолжает остальных. Одна проблемная страница не должна блокировать всю рассылку.
- **Ошибка резолва screen_name** — лид помечается ⚠️ ещё на экране подтверждения, не участвует в отправке.
- **Закрытие вкладки/обновление страницы во время рассылки** — отправка прерывается на середине без возможности возобновления. Это осознанное ограничение MVP (личный ручной инструмент, невысокие объёмы) — не стоит усложнения persistent job queue ради него сейчас.
- **Дневной лимит** — предупреждение, не блокировка (см. выше).

## Тестирование

В проекте нет автоматических тестов ни для `index.html`, ни для edge functions (существующая практика). Проверка — вручную:
1. Резолв: лид с screen_name-ссылкой без `vkPeerId` → после открытия экрана подтверждения должен получить числовой peer_id и попасть в список ✅
2. Лид с приватным/несуществующим профилем → должен остаться ⚠️ и не попасть в отправку
3. Рассылка на 2 тестовых получателя (свои тестовые VK-страницы, не реальные лиды) — оба сообщения приходят, оба логируются в историю диалога как обычные исходящие
4. Принудительная ошибка у одного получателя (например, невалидный peer_id) — рассылка продолжает второго получателя, первый помечен ❌
5. Кнопка «Остановить» на середине — оставшиеся получатели остаются «в очереди», уже отправленные не откатываются
6. Баннер дневного лимита появляется при приближении к 30-50, но не блокирует кнопку «Начать рассылку»

## Вне рамок (Out of Scope)

- Планирование рассылки на будущее время
- Возобновление прерванной рассылки после перезагрузки страницы
- Отдельный пул шаблонов специально для рассылок (переиспользуем существующие)
- Автоматическая блокировка при превышении дневного лимита (только предупреждение)
